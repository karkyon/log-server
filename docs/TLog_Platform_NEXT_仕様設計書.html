SECTION 01

コンセプト・解決したい課題

🎯 ONE LINE: どんなWebアプリにも1ファイル組み込むだけで、ユーザーの操作ログを自動収集・蓄積し、 セッション単位でタイムライン分析・チケット発行できるSaaS型ログ管理CMSを構築する。

現在の問題点

⚠️ TALON専用になっている
ロガー（talon_testcase_logger.js）がTALON固有の実装に依存しており、他のWebアプリには流用しにくい構造になっている。
⚠️ GitHubバッチ処理モデル
ログ収集→GitHub Push→Actions実行→静的HTML生成というフローは、リアルタイム性がなく、CMSとしての操作性がない。
⚠️ プロジェクト単体管理
1プロジェクト前提の設計。複数アプリのログを横断管理する仕組みがない。
⚠️ TraceIDが不明確
現状のTraceIDはランダム生成で、「ユーザーの1セッション」という意味が付与されていない。ログイン/ログアウトとの連動がない。

実現したい世界

📦 SDK化（汎用ロガー）
1行のscript importで、TALONでも社内Webシステムでも将来のNext.jsアプリでも同じ方法でログ収集を開始できる。
🔐 CMS（ログ管理Webアプリ）
ユーザー認証付きのWebダッシュボード。プロジェクト選択→ログ蓄積確認→TraceID単位分析→チケット発行まで一気通貫。
🔄 リアルタイム蓄積
GitHubバッチ処理ではなく、ログはAPIサーバーに直接蓄積。CMSから即座に最新ログを参照できる。

SECTION 03

TraceID（トレースID）設計 ★最重要コンセプト

TraceIDの定義： 監視対象Webアプリ（TALONや将来のNext.jsアプリなど）において、 あるユーザーがログインしてからログアウトするまでの1操作セッションを一意に識別するID。 ログアウト（またはセッション切断）時に自動リセットされ、次のログイン時に新しいIDが発行される。

TraceIDのライフサイクル

① 監視対象アプリ
ログイン（ユーザー認証完了）
→
② SDK が TraceID 発行
proj_abc:user_xyz:2026-03-02T09:15:00:r4nd
→
③ 以降の全ログに
TraceID付与（SCREEN_LOAD, UI_CLICK等）
↓ ユーザー操作が続く間、同じ TraceID
④ ログアウト or セッション切断
（タイムアウト含む）
→
⑤ TRACE_END ログ送信
+ TraceID クリア
→
⑥ 次回ログイン時
新しいTraceID発行

TraceIDの構造

// TraceID フォーマット // {projectSlug}:{userId}:{ISO日時}:{ランダム4文字} "machining-sys:user_42:20260302T091500:a9f2" // CMSでの表示用（短縮形） "TR-20260302-001" // プロジェクト内の日次連番

TraceIDが持つメタデータ

フィールド型説明例
traceIdString一意識別子（フル形式）machining-sys:user_42:20260302T091500:a9f2
projectIdIntどのプロジェクト（アプリ）か3
userIdString?監視対象アプリのユーザーID（任意）"user_42" or null
userLabelString?表示名（ログから推測）"大野 智"
startedAtDateTimeセッション開始時刻2026-03-02T09:15:00Z
endedAtDateTime?セッション終了時刻（null=継続中）2026-03-02T11:32:44Z
durationInt?セッション継続秒数8264
screenCountInt閲覧した画面数12
clickCountIntクリック操作回数87
errorCountIntエラー発生回数3
statusStringACTIVE / CLOSED / TIMEOUTCLOSED

TraceIDのリセットトリガー

✅ 自動リセット（推奨）
SDK が監視対象アプリのログアウトイベントを検知して自動的にリセット。 SDKの初期化時に loginEvent / logoutEvent のCSSセレクタを指定するか、 監視対象アプリが SDK の API を明示的に呼び出す。
🔄 その他のリセット条件
タイムアウト（デフォルト：無操作30分）、ページ離脱（beforeunload）、 セッションストレージのクリア検知、SDK.resetTrace() の明示呼び出し

SECTION 05

TLog SDK 設計（汎用ロガー）

組み込み方法（3パターン）

① CDN（最速）
<script src="https://{tlog-host}/sdk/tlog-sdk.js" data-project="machining-sys" data-api-key="ak_xxx"> </script>
HTMLに1行追加するだけ。TALONやレガシーシステムに最適。
② npm（モダンアプリ）
import { TLog } from '@tlog/sdk';
TLog.init({ project: 'my-app', apiKey: 'ak_xxx' });
Next.js / React / Viteアプリに最適。TypeScript型定義付き。
③ 手動注入（TALON継承）
現行の talon_testcase_logger.js をtlog-sdk.jsに置き換え。 既存のTLog.screenLoad() / TLogAutoInstrument.init() APIは互換維持。

SDK 初期化オプション

TLog.init({ // 必須 project: "machining-sys", // CMS登録済みプロジェクトslug apiKey: "ak_xxxxxxxxxxxxx", // CMS管理画面で発行するAPIキー endpoint: "http://192.168.1.11:3099", // TraceID制御（★NEXTの肝） traceMode: "session", // "session" | "manual" | "page" loginEvent: ".login-success", // ログイン完了を示すDOMイベント or CSSセレクタ logoutEvent:".logout-btn", // ログアウトボタンのCSSセレクタ sessionTimeout: 1800, // 無操作タイムアウト（秒）デフォルト30分 // ユーザー情報取得（監視対象アプリのユーザーをTraceIDに紐付け） getUserId: () => window.currentUser?.id, // ← アプリ側で実装 getUserLabel: () => window.currentUser?.name, // 収集設定 screenshot: { enabled: true, onError: true, onClick: true }, console: { enabled: true, levels: ["error", "warn"] }, autoInstrument: true, // ボタン等を自動計装 });

イベント種別説明旧TLogとの対応

APIキー認証フロー

SDKが送信するHTTPリクエストには Authorization: Bearer ak_xxx ヘッダーを付与。 APIキーはプロジェクト単位でCMS管理画面から発行する。 APIキーが無効・プロジェクトが非アクティブの場合、サーバーは403を返しSDKはIndexedDBにキューイング。 APIキーにはIP制限・レートリミットを設定可能。
SECTION 06

TLog CMS 機能設計

CMSのユーザー種別（CMSにログインする人）

⚠️ 用語の整理：「CMSユーザー」＝TLog CMSにログインする開発者・テスター。 「監視対象アプリユーザー」＝TALONなどを実際に使う現場ユーザー。両者は別物。
ロールできること
ADMIN全操作 + ユーザー管理 + プロジェクト管理 + APIキー発行
MEMBER参加プロジェクトのログ閲覧 + TraceID分析 + チケット発行・編集
VIEWER参加プロジェクトのログ閲覧のみ（編集不可）

主要画面・機能

🏠 ダッシュボード（ログイン後トップ）
参加プロジェクト一覧・各プロジェクトの直近ログ件数・アクティブTraceID数・ 未解決チケット数をカード表示。プロジェクトを選択して詳細へ遷移。
📋 TraceID一覧（プロジェクト選択後）
日付フィルタ・ユーザーフィルタ・ステータスフィルタ付きのTraceID一覧。
各行に表示：TraceID（短縮形）/ 監視対象ユーザーID / セッション開始・終了時刻 / 継続時間 / 操作画面数 / クリック数 / エラー数 / ステータス（ACTIVE/CLOSED/TIMEOUT）
TraceIDをクリックすると詳細（タイムライン）へ遷移。
⏱️ TraceIDタイムライン（詳細画面）★核心機能
選択したTraceID内の全操作を時系列表示。
表示内容： SCREEN_LOAD（画面名）→ UI_CLICK（ボタン名）→ BACKEND_REQ/RES → ERRORの流れをタイムライン形式で表示。 各ステップにスクリーンショットサムネイル・コンソールログを紐付け表示。
操作： ステップ選択→チケット発行。複数ステップ選択→作業パターンとして保存。
📊 作業パターン分析
複数のTraceIDから共通の操作シーケンスを抽出してパターン登録。 パターン名・説明・OK/NG判定・メモを付与。 フロー図（SVGセルペンタイン）で視覚化。
🎫 チケット管理（Issue Tracker）
TraceIDタイムラインから直接発行： 対象ステップのスクリーンショット・ログが自動添付。 チケット種別（不具合/仕様違い/改善提案）/ 優先度（高/中/低）/ 担当者 / ステータス管理。 将来的にJiraやGitHub Issuesとの連携も想定（Webhook）。
⚙️ プロジェクト管理（ADMIN）
プロジェクト作成・編集・メンバー招待・APIキー発行（複数キー対応・有効期限付き）。 ログ保持期間設定・スクリーンショット保存上限設定。

SECTION 08

API設計

認証の二層構造（★NEXT版の重要設計）

🔑 SDKからの通信（APIキー認証）
監視対象アプリのブラウザ → TLog API Server
Authorization: Bearer ak_xxxxxxxxx
APIキーはプロジェクト単位。ログ送信専用エンドポイントのみアクセス可。 読み取り・管理操作は不可。
🔑 CMS からの通信（JWT認証）
CMSユーザーのブラウザ → TLog API Server
Authorization: Bearer jwt_token
CMSログイン後に発行。アクセス可能なプロジェクトはProjectMembershipで制御。
